# ADR 001: Typed SSE Event Models and Loose Edge Payloads

*Date:* 2026-06-10
*Status:* proposed

## Context

`codex-proxy-rs` was scaffolded specifically to de-risk the hardest part of the
Go → Rust port: the stateful SSE transformer that rewrites upstream Codex
streaming events into OpenAI chat-completion chunk JSON. The first scaffold pass
kept `src/transform.rs` intentionally simple and used `serde_json::Value` plus
ad hoc extraction for the first implemented event paths:

- `response.created`
- `response.output_text.delta`
- `response.completed`
- `[DONE]`

That was the right bootstrap move for getting a compiling crate, a fenix-backed
shell, and the first ported Go tests (`tests/transform_sse.rs`) green quickly.
But the next phase is broader parity with the Go transformer in
`/Users/jack/git/github.com/jmmaloney4/codex-proxy/internal/server/transform.go`,
including reasoning events, tool-call start events, tool-call argument deltas,
and final completion behavior. Leaving the whole upstream event surface as raw
`Value` would keep the Rust port overly stringly-typed in the exact area where
we want confidence from the compiler.

At the same time, the upstream SSE payload family is not uniformly clean. Some
fields are stable and semantically load-bearing (`type`, `sequence_number`,
`response.id`, usage fields, tool IDs, text deltas). Other parts — especially
reasoning payload internals and any still-unported event families — are more
variant-heavy, less proven by tests, and not worth over-modeling before we have
ported the full Go test corpus.

The architectural question is therefore not "typed structs or `Value`" in the
abstract. It is where the schema boundary should live in this port so that:

1. stable event/output shapes are typed,
2. unstable or low-value payload edges remain flexible,
3. test-porting work stays incremental rather than blocked on full schema
   archaeology.

## Decision

We will adopt a **typed core, loose edges** model for the SSE transformer.

1. **Typed serde models for handled upstream event variants.**
   The transformer will stop indexing directly into `serde_json::Value` for the
   stable event families it actively handles. Instead, it will deserialize those
   events into small typed structs/enums keyed by the upstream `type` field.

2. **Typed serde models for emitted OpenAI chunk JSON.**
   The outbound chunk objects generated by `SSETransformer` will be represented
   as explicit Rust `Serialize` structs rather than constructed ad hoc with
   `json!` blobs.

3. **Keep unstable or not-yet-needed upstream edges loose.**
   Reasoning payload internals, unused nested fields on tool-call items, and
   unknown/unhandled event families may remain partially typed or raw
   `serde_json::Value` behind narrow boundaries until the corresponding Go tests
   are ported and the shape is proven stable.

4. **Refactor in phases, not in one schema rewrite.**
   The first typed refactor pass covers the event families already implemented in
   `src/transform.rs`: `response.created`, `response.output_text.delta`, and
   `response.completed`. Reasoning and tool-call families follow only when their
   test slices are being ported.

## Alternatives Considered

1. **Keep the entire transformer `Value`-driven** – Lowest immediate effort, but
   it preserves stringly-typed access in the exact code path we are using as the
   correctness anchor for the rewrite. It would make the parity phase harder to
   review and easier to break subtly.

2. **Model the full upstream SSE schema rigidly up front** – Attractive from a
   type-purity perspective, but too expensive and too speculative for the current
   port stage. It would front-load work on reasoning/tool payload shapes before
   the Go tests prove which fields actually matter.

3. **Typed core with loose edges (chosen)** – Type the stable, semantically
   important event and output shapes now; leave unstable or still-unported
   payload interiors flexible until their behavior is grounded by tests.

## Consequences

- **Pros:**
  - The handled transformer paths move from string-key lookups to compiler-checked
    serde models.
  - The OpenAI chunk schema becomes explicit Rust code rather than scattered
    inline JSON construction.
  - Future Go test ports should get easier because event families can be added as
    small typed variants instead of by expanding a single unstructured blob.
  - The design preserves incremental delivery: we do not have to discover and
    encode every upstream payload variant before shipping useful parity work.

- **Cons:**
  - The transformer will temporarily live in a hybrid state, with typed handled
    variants alongside looser fallback parsing for unported event families.
  - Reasoning and tool-call payload modeling decisions are deferred rather than
    resolved once, so there will be a second round of design work when those
    tests are ported.
  - The repo gains more schema types and module boundaries earlier than a pure
    prototype would otherwise require.

## Technical Details

### Typed immediately

These objects/responses should be represented as real serde structs now:

- **Handled upstream event envelope/variants**
  - `response.created`
  - `response.output_text.delta`
  - `response.completed`
- **Stable upstream nested objects/fields**
  - `response.id`
  - `response.usage`
  - text-delta fields: `delta`, `item_id`, `output_index`, `content_index`
- **Outbound OpenAI chunk objects**
  - chat completion chunk root
  - choice objects
  - delta objects
  - usage object
- **Internal transformer state**
  - keep the existing typed `SSETransformer` fields as-is

### Partially typed initially

These shapes should be typed only around the fields we actively use:

- **Reasoning event families**
  - type `type`, `sequence_number`, `output_index`
  - keep nested reasoning-content payloads loose until the full reasoning test
    slice is ported
- **Tool-call event families**
  - type `item_id`, `delta`, `item.id`, `item.call_id`, `item.name`, `item.type`
  - keep unused nested item metadata loose initially

### Left loose for now

These remain `serde_json::Value` or a narrow fallback wrapper for the moment:

- unknown/unhandled upstream event families
- nested payload details not yet needed by the Rust implementation or tests
- future upstream additions we want to ignore safely until explicitly adopted

### Recommended module boundary

The transformer should be split into three layers:

1. **Upstream parse boundary**
   - small typed input structs/enums for handled event variants
2. **State machine**
   - `SSETransformer` owns stream state and variant dispatch
3. **Outbound emit boundary**
   - typed OpenAI chunk structs serialized to JSON bytes

A likely file layout is:

- `src/transform.rs` — orchestration/state machine
- `src/transform/upstream.rs` — typed input event models
- `src/transform/openai.rs` — typed output chunk models

This split is preferred, but the first refactor pass may keep the types nested in
`src/transform.rs` if that keeps churn smaller while the crate is still tiny.

## Risks & Mitigations

| Risk | Mitigation |
| ---- | ---------- |
| We over-model an unstable upstream payload family and spend time undoing it later. | Limit the first pass to `response.created`, `response.output_text.delta`, and `response.completed`. Only tighten reasoning/tool schemas when their Go test slices are actively being ported. |
| A single giant serde enum becomes awkward to maintain as event families expand. | Use a minimal event probe or a small tagged enum for handled variants only, with a loose fallback path for unknown/unhandled events. |
| The hybrid typed/loose state becomes permanent and messy. | Make the phased rollout explicit in code comments and in the test-porting plan; each new Go test slice should either add a typed variant or justify staying loose. |
| Output-schema refactors accidentally change wire behavior. | Keep the existing ported tests as a guardrail and add more parity tests before broadening the typed output surface. Compare parsed JSON rather than hand-waved string output. |

| Task | Estimate | Notes |

## Appendices

### Appendix A: Decision boundary in one sentence

Use typed serde structs for the stable event discriminator, handled upstream
payloads, and emitted OpenAI chunk objects; use partial structs plus
`serde_json::Value` only at the unstable or not-yet-worth-modeling edges.

### Appendix B: Why not wait until full parity to type anything?

Because the whole point of the Rust port is to move the hardest transform logic
into a form the compiler can help us with. If we postpone typed models until
after the full parity pass, we will spend the highest-risk phase expanding a
stringly-typed transformer rather than shrinking it. The better sequence is:
bootstrap quickly with `Value`, then type the core as soon as the first tests are
proving the path.
